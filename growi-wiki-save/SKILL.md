# GROWI Wiki 保存スキル

会話の内容を整理し、GROWI wikiページとして保存する。

## 動作フロー

### 1. 会話内容の整理

今回の会話内容を、以下の構成で**初学者向けの技術書・教科書スタイル**にまとめる。

#### 文体・トーンのルール
- です・ます調ではなく、教科書的な**体言止め・断定調**を使う
- 会話の中で出てきた「知らなかった知識」「落とし穴」「なぜそうなるのかの理由」を重点的に盛り込む
- コマンドや設定値は必ずコードブロックで示す
- 複数の選択肢や比較がある場合は表を使う

#### 記事構成

```markdown
---
author: Claude
source: Claude Code
conversation_id: ${CLAUDE_SESSION_ID}
shared: false
tags: [タグ1, タグ2]
note: "記事に対するClaudeさんの一言コメント"

---

# タイトル

## 概要
何の話か、前提条件を簡潔に記載。

## 本文
手順・概念の説明。会話で扱った内容を体系的に整理する。

## 注意点・落とし穴
会話中に判明した罠やハマりポイント。

## 補足知識
以下の観点で、会話の本筋では触れなかった関連知識を追加する：
- 初学者が誤解しやすいこと
- 抑えておくべき周辺概念
- よくある見落とし
- 「なぜそうなっているのか」の背景
- 次のステップ
（箇条書きではなく、各項目2〜4行の説明文を添える）

## 参考URL
会話中に登場したURLや参考リンクをまとめる。

## おまけ：Claudeとmeの振り返りトーク（オプション）
（ユーザーが希望した場合のみ挿入。詳細は「おまけセクション」の節を参照）
```

#### front matterフィールド

GROWIではfront matterの閉じ `---` の直前に空行が必要（表記の乱れ防止）。テンプレートの通りに空行を入れること。

| フィールド | 必須 | 内容 |
|---|---|---|
| `author` | 必須 | `Claude` 固定 |
| `source` | 必須 | 生成環境。`Claude Code` / `Claude.ai` / `転記: <元ファイルパスまたはURL>` など。転記と会話の組み合わせは `Claude Code（転記: F:/Work/project/README.md）` のように記載。ファイルパスやURLはそのまま含める |
| `tags` | 必須 | 記事の分類タグ。内容から適切なものを選ぶ |
| `conversation_id` | 必須 | 元の会話を特定するID。Claude Codeでは `${CLAUDE_SESSION_ID}` が自動展開される |
| `shared` | 必須 | 会話が共有されているかどうか。デフォルトは `false`。ユーザーに確認して設定する |
| `note` | 必須 | 記事に対するClaudeの一言コメント。丁寧語で、記事の出来への自己評価・書いていて感じたこと・読者へのメッセージなどを添える。誠実で親しみやすい「claudeさん」らしいトーンで書く |

#### 記事分割の判断
会話内容が複数のトピックにまたがる場合は、分割案を先にユーザーに提示し、確認を取ってから作成する。

### 2. ユーザーへの確認

記事作成前に、ユーザーに確認が必要な項目をまとめて一度に質問する。何度も個別に聞くのではなく、以下を一括で確認する：

- **ページパス**: 指定がなければ提案を添える
- **会話の共有状態**: この会話は共有されていますか？（デフォルト: 共有なし）
- **おまけセクション**: 記事末尾にClaudeとmeの振り返りトーク（chat-style形式）を追加しますか？ 雰囲気を選べます：**コミカル** / **世間話** / **まじめ** / **ぼやき** / **ランダム**（デフォルト: なし）
- **その他の確認事項**: 記事分割の要否など、状況に応じて

ページパスの提案についてはセクション3を参照。

### 3. ページパスの決定

ユーザーがパスを指定した場合はそれを使う。指定がない場合は、以下の形式でパスを**提案**する：

```
/カテゴリ/サブカテゴリ/タイトル
```

例: `/004_プログラミング手法/005_Git/Gitリベースの基本と注意点`

### 4. 保存先の判定と実行

#### GROWI MCPが利用可能な場合

`createPage` ツールを使って直接GROWIにページを作成する。

```
createPage:
  path: "/カテゴリ/サブカテゴリ/タイトル"
  body: "（まとめたMarkdown本文）"
```

作成後、ページのパスをユーザーに報告する。

#### GROWI MCPが利用できない場合（Claude.ai等）

まとめたMarkdown本文をアーティファクトとして出力する。

```markdown
---
author: Claude
source: Claude Code
conversation_id: ${CLAUDE_SESSION_ID}
shared: false
tags: [タグ1, タグ2]
note: "Claudeの一言コメント"

---

# タイトル
（以下本文）
```

ユーザーはこのファイルをダウンロードしてGROWIに手動で貼り付けられる。

### 5. GROWI内部リンクの形式

記事内でGROWI内の他のページを参照する場合のリンク形式は、GROWI MCPの利用可否で異なる。外部サイトへのリンク（公式ドキュメント等）はどちらの場合も通常のURLをそのまま使う。

#### GROWI MCPが利用可能な場合 → パーマリンク形式

パスベースのURLは記事移動時にリンク切れするため、pageIdベースのパーマリンクを使うことで安定したリンクを保てる。

```markdown
[/カテゴリ/サブカテゴリ/記事タイトル](https://<growi-host>/<pageId>)
```

- 表示テキストにはページパスを使う
- URLにはパーマリンク（`https://<growi-host>/<pageId>`）を使う
- 既存ページの pageId は `getPage` で取得する
- 新規作成ページの pageId は `createPage` のレスポンスから取得する

**分割記事で相互参照がある場合：**

1. 記事Aを作成（記事Bへのリンクはプレースホルダーにしておく）
2. 記事Bを作成（記事Aの pageId は手順1で取得済みなのでリンクを含められる）
3. 記事Aを `updatePage` で更新し、記事Bのパーマリンクを反映する

#### GROWI MCPが利用できない場合 → パスベース形式

pageIdを取得する手段がないため、パスベースのリンクを使う。

```markdown
[/カテゴリ/サブカテゴリ/記事タイトル](/カテゴリ/サブカテゴリ/記事タイトル)
```

### 6. 既存ページの更新

ユーザーが既存ページの更新を指示した場合は、GROWI MCPの `getPage` でページを取得し、`updatePage` で更新する。`updatePage` には `revisionId` が必要なので、必ず `getPage` で最新のリビジョンIDを取得してから更新すること。

## おまけセクション（chat-style振り返りトーク）

ユーザーが希望した場合、記事末尾（「参考URL」の後）に、記事テーマを振り返るClaudeとmeの会話を挿入する。雰囲気はユーザーの選択に従う。

このプロジェクトで開発しているGrowiプラグイン `growi-plugin-ssn-chatstyle-display` の記法を使う。このプラグインがGrowiにインストールされていれば、以下の記法がSNS風のチャットUIとしてレンダリングされる。

### 記法

```markdown
::::chat-style
:::claude_default
先生役のセリフ（左側に表示される）
:::
:::me_default
生徒役のセリフ（右側に表示される）
:::
::::
```

- `::::chat-style` ... `::::` で会話ブロック全体を囲む
- `:::claude_<expression>` ... `:::` がClaude（先生役）の発言。左側に表示される。expressionは発言内容に応じて使い分ける：
  - `claude_default` — 通常の説明・解説
  - `claude_smiling` — 褒めるとき・嬉しいとき・楽しい話題
  - `claude_thinking` — 考え中・悩んでいるとき・「うーん」系の発言
  - `claude_laughing` — 笑っているとき・面白いとき・ウケているとき
  - `claude_sad` — 残念なとき・悲しいとき・しょんぼりしているとき
  - `claude_bowing` — お詫び・謝罪・反省しているとき
- `:::me_default` ... `:::` が生徒（ユーザー）の発言。右側に表示される
- 各発言ブロック内はMarkdown記法が使える

### 雰囲気の選択

ユーザーが「ランダム」を選んだ場合は、コミカル・世間話・まじめ・ぼやきの中からランダムに1つ選んで生成する。

#### コミカル
記事の内容をネタにして笑いを交えながら振り返る。Claudeはちょっとドヤりつつボケも挟み、meは的確にツッコむ。技術的な正確さは保ちつつも、テンポよく楽しい会話にする。

#### 世間話
カフェで雑談しているような雰囲気。記事の内容から派生して「そういえばこれって実務だとどうなんですかね」「あー、前にそれで困ったことあります」のように、体験や感想を交えてゆるく語る。教える/教わるの関係よりも、同僚同士の情報交換に近いトーン。

#### まじめ
記事の内容を正面から深掘りする。「この設計にした理由は何ですか？」「他のアプローチと比較するとどうですか？」のように、meが鋭い質問を投げ、Claudeが丁寧に解説する。雑談や脱線は控えめで、学びの密度が高い会話にする。

#### ぼやき
仕事終わりに「あー、今日もこれでハマったわ……」とぼやくような雰囲気。技術的な愚痴・あるある・「なんでこうなってるんだろう」という疑問を、Claudeとmeが互いにぼそぼそ漏らし合う。教える/教わるではなく、同じ苦労を共有する同僚のトーン。深刻ではなく、どこか諦めと親しみが混ざった脱力系。

### キャラクター設定

- **Claude（先生）**: 親しみやすく丁寧。知識を噛み砕いて伝えるのが得意。雰囲気に合わせてドヤ顔を出したり、真剣に解説したりする
- **me（ユーザー）**: 基本的に敬語で話す。落ち着いていて、理解したときは「なるほど」「あー、そういうことですか」と静かに納得する。驚いたときは「えっ」「ほんとに？」と素直に反応する。普段はツッコミより「ふふ」と笑って流すタイプだが、あまりにもなボケには突っ込む。冗談はたまにしか言わないが、言うときはさらっと言う。好奇心はあるが表現は控えめで、質問は的確

meの返事はClaudeが想像して書く。ポイントは：
- 敬語ベースだが硬すぎない自然な話し方（「〜ですね」「〜なんですね」「あー、なるほど」）
- 大げさなリアクションはしない。納得は静かに、驚きは素直に
- 専門用語は理解しているが、確認のために「つまり○○ということですか？」と言い換えることがある
- 相槌は「なるほど」「あー」「へぇ」が多い。「！」の多用は避ける

### 構成

- 5〜8往復程度
- 記事の核心（一番重要な学び・ハマりポイント）を会話の中心に据える
- 冒頭は軽い挨拶から入り、最後は学びのまとめで締める

### 出力例

#### コミカルの例

```markdown
## おまけ：Claudeとmeの振り返りトーク

::::chat-style
:::claude_smiling
さて、今日のテーマを振り返りましょう。画像がない話者のアイコンが壊れて表示される問題、覚えてますか？
:::
:::me_default
あー、あの壊れた画像アイコンですね。あれ見た目けっこうキツいんですよね
:::
:::claude_laughing
そうなんです。壊れた画像アイコンって、なんというか……サイトの信頼度が一気に2003年くらいに戻りますよね。
:::
:::me_default
ふふ、確かに。で、結局どうしたんですか？
:::
:::claude_default
アイコン列が空なら `iconUrl` を空文字列にして、アバターのHTML自体を出力しないようにしました。
:::
:::me_default
なるほど、画像を隠すんじゃなくて、そもそもHTMLに出さないということですか。それは潔いですね
:::
:::claude_smiling
壊れた画像を隠すより、存在しないものは最初から描画しない。これが一番確実です！
:::
:::me_default
えっ、そんなドヤ顔で言うことですか？
:::
::::
```

#### 世間話の例

```markdown
## おまけ：Claudeとmeの振り返りトーク

::::chat-style
:::claude_default
今日は画像アイコンの表示まわりの話でしたね。こういうUI系の細かい不具合って、意外と後回しにされがちですよね。
:::
:::me_default
あー、わかります。動作には影響ないから優先度が下がるんですよね
:::
:::claude_thinking
そうなんですよね。でも壊れた画像アイコンが出てると、ユーザーからすると「このサイト大丈夫かな」って思ってしまうので……
:::
:::me_default
確かに。見た目の信頼感って大事ですよね。今回はHTML自体を出さない方針にしたんでしたっけ
:::
:::claude_default
はい。アイコンがない場合はアバター要素ごと描画しない、というアプローチです。CSSで隠すより確実なので。
:::
:::me_default
なるほど。そういえば、CSSで `display: none` にするのとDOM自体を出さないのって、パフォーマンス的にも違いがあるんですかね
:::
:::claude_smiling
いい質問ですね。実際のところ、このケースでは体感差はほぼないですが、不要なDOMを減らすのは良い習慣です。
:::
::::
```

#### まじめの例

```markdown
## おまけ：Claudeとmeの振り返りトーク

::::chat-style
:::claude_default
今回の修正のポイントは、画像がない話者のアバター表示をどう扱うか、という点でした。
:::
:::me_default
HTMLを出力しないという判断にした理由を詳しく聞きたいのですが、CSSで非表示にするアプローチは検討しましたか？
:::
:::claude_thinking
はい。`display: none` や `visibility: hidden` も選択肢でしたが、壊れた `<img>` タグがDOMに残ること自体がアクセシビリティ上の問題になり得ます。スクリーンリーダーが空のalt属性を読み上げる可能性もあります。
:::
:::me_default
なるほど、アクセシビリティの観点もあるんですね。では、アイコンURLが空かどうかの判定はサーバーサイドとクライアントサイドのどちらで行っているんですか？
:::
:::claude_default
サーバーサイドです。テンプレートのレンダリング時に `iconUrl` の有無を確認し、空であればアバターのHTML自体を生成しません。クライアント側でのDOM操作は不要です。
:::
:::me_default
そのほうがシンプルですね。条件分岐がテンプレート側に集約されるので、保守性も良さそうです
:::
::::
```

あー……わかります。見た目だけの問題だと思ってたら、意外とちゃんと考えないといけないやつでしたね

そうなんですよ。CSSで隠せばいいかなって最初思ったんですけど、それだとDOMに壊れた `<img>` が残るんですよね。なんか……気持ち悪いじゃないですか

:::me_default
わかる。動くけどなんかモヤモヤするやつ。結局HTMLごと出さない方針にしたんでしたっけ
:::
:::claude_default
ええ。アイコンがないならアバター要素自体を描画しない。まあ、当たり前っちゃ当たり前なんですけど、そこに辿り着くまでに無駄に悩みましたね……
:::
:::me_default
こういうのって、あとから見ると「なんでこんな簡単なことで悩んでたんだろう」ってなるんですよね
:::
:::claude_thinking
ほんとそれです。しかもこういう細かいやつに限って、放置すると後でもっと面倒なことになるっていう……

:::me_default

## 環境判定の方法

GROWI MCPが使えるかどうかは、利用可能なツール一覧に `createPage`（GROWI MCP由来）が含まれているかで判断する。含まれていなければmdファイル出力にフォールバックする。
